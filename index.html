The Inherent Logic of Explicit Correctness

How type-driven development eliminates accidental complexity by encoding domain invariants into the type system

Part 1

Theoretical Foundations

Take some time to familiarize yourself with the theory behind the idea I want to express.

1. Primitive Obsession

"Primitive Obsession is the use of primitive data types to represent domain concepts." Martin Fowler, Refactoring: Improving the Design of Existing Code, 1999

The code smell describes the habit of using string, int, float everywhere instead of creating small domain-specific types. An age, an income, a child count, and a postal code are all integers, but they are not the same thing. Treating them as raw integers forces every consumer to remember and re-implement the rules that distinguish them.

2. Make Implicit Concepts Explicit

"Many transformations of domain models and the corresponding code happen when developers recognize a concept that has been hinted at in discussion or present implicitly in the design, and they then represent it explicitly in the model with one or more objects or relationships." Eric Evans, Domain-Driven Design: Tackling Complexity in the Heart of Software, Chapter 9, 2003

If a concept exists in the domain language but is not represented as a first-class object in the code, it is implicit. Making it explicit means giving it a name, a type, a constructor, and a home for its invariants.

3. The Three Causes of Complexity (Out of the Tar Pit)

Ben Moseley and Peter Marks (2006) identify three causes of complexity in software systems.

3.1 Complexity Caused by State

"The single biggest remaining cause of complexity in most contemporary large systems is state, and the more we can do to limit and manage state, the better." Moseley & Marks, Out of the Tar Pit, 2006

Mutable state makes systems unpredictable: a test on a system in one state tells you nothing about its behaviour in a different state. Every added bit of state doubles the number of possible states the system can be in.

3.2 Complexity Caused by Control

"Control is basically about the order in which things happen. When a programmer is forced to specify the control, he or she is being forced to specify an aspect of how the system should work rather than simply what is desired. Effectively they are being forced to over-specify the problem." Moseley & Marks, Out of the Tar Pit, 2006

Imperative code forces you to state execution order even when you only care about the relationship between values. Every conditional guard, every nested validation block, is accidental control flow.

3.3 Complexity Caused by Code Volume

"Many of the classic problems of developing software products derive from this essential complexity and its nonlinear increase with size." Fred Brooks, cited in Out of the Tar Pit, 2006

Code volume is largely a secondary effect of the first two causes. Complexity grows nonlinearly with size.

"...we have to keep it crisp, disentangled, and simple if we refuse to be crushed by the complexities of our own making..." Edsger W. Dijkstra, EWD1243, 1997

4. Type-Driven Development

"Make illegal states unrepresentable." Yaron Minsky, Harvard guest lecture, 2010

The approach was formally developed in Edwin Brady's Type-Driven Development with Idris (Manning, 2016). The core idea: encode domain constraints into types at construction time, so any value that exists in the system is already valid by definition. Applied to function signatures, the signature itself becomes documentation.

Part 2

The Theory

A strongly typed flow, in which a request transitions through a sequence of value-typed stages (a type-driven state machine), simultaneously addresses all three causes of complexity:

  • State complexity is reduced because the state of the request is encoded in its type. You do not need to store or manage intermediate validation flags; the type at each stage is the state.
  • Control complexity is reduced because conditional validation guards disappear from function bodies. Construction either succeeds or throws. The function that receives a well-typed argument has no branching to perform for correctness.
  • Code volume is reduced because validation is written once, inside the value object constructor, and reused everywhere the type appears.

Furthermore, this principle generalises beyond APIs. Functions, services, mobile app flows, website page transitions: any system with identifiable stages can be modelled as a type-driven state machine where each stage is a distinct type.

Better. But every function that receives product_name, quantity, unit_price, or discount_percent duplicates this code. The complexity caused by control has grown: we added four conditional branches that have nothing to do with the line item calculation itself.

Step 3: Extract Assertions

Validation is now reusable. But look at the function signature: (string, int, float, int). It still communicates nothing. An architectural diagram of this function reveals no domain concepts: four primitives go in, a float comes out.

Every domain concept now has a name, a home, and a visible contract.

Step 4: Value Objects

With value objects, validation lives in the constructor. The function body is now a single expression with no branches at all.

  • State: there is no hidden validation state; a value either exists as a valid type or it does not exist at all.
  • Control: the function body has no branches at all; every guard has been removed by the type system.
  • Code volume: the function is shorter; the intent is clearer.
  • Signature as documentation: the function tells you exactly what domain concepts it requires.

Type-Driven API Design

stateDiagram-v2
    direction TB
    [*] --> IncomingRequest : HTTP GET /cart/line-total
    IncomingRequest --> HTTP401 : firewall, user not authenticated
    state "Authenticated Request" as AuthenticatedRequest {
        a1 : session user
    }
    IncomingRequest --> AuthenticatedRequest : firewall passes

        c1 : ProductName.fromString()
        c2 : Quantity.fromInt()
        c3 : Price.fromFloat()
        c4 : DiscountPercent.fromInt()
    }
    AuthenticatedRequest --> Construction : framework dispatcher
    Construction --> HTTP422 : any constructor throws, invalid domain value
    state "Validated Line Item Request" as ValidatedRequest {
        v1 : ProductName name
        v2 : Quantity quantity
        v3 : Price unitPrice
        v4 : DiscountPercent discount
    }
    Construction --> ValidatedRequest : all types constructed
    state "Line Item Result" as LineItemResult {
        r1 : float total
    }
    ValidatedRequest --> LineItemResult : calculate_line_total(), no guards needed
    LineItemResult --> [*] : HTTP 200
    HTTP401 --> [*]
    HTTP422 --> [*]

Each arrow is a transition that either succeeds (producing a richer type) or fails (producing an HTTP error). By the time calculate_line_total() is reached, every guarantee has already been enforced structurally.

Before: Naive Controller

In a naive implementation, the controller absorbs all the complexity we extracted into value objects. A request like GET /cart/line-total?product_name=Widget&quantity=2&unit_price=9.99&discount_percent=10 arrives as raw strings and the controller must validate everything itself.

After: Type-Driven Controller

The framework dispatcher automatically deserializes raw query parameters into value objects before the controller action is called. You register the mapping rule once, and every action that declares a value object as a parameter benefits automatically. Try GET /cart/line-total?product_name=Widget&quantity=0&unit_price=9.99&discount_percent=10: the framework returns a 422 Unprocessable Entity before your action body runs. The value 0 is a valid integer, but it fails the domain rule inside the Quantity constructor. No controller code handles this.

"The framework becomes the enforcement layer. You write the rule once (in the value object), and the entire application surface area enforces it."

Part 6

Type-Driven State Machines

A shopping cart has two structurally distinct states: an empty cart and a cart with items. These are not the same type. Operations that are invalid for a given state do not need runtime guards; the type itself makes them impossible or explicitly handled. remove_item called on an EmptyCart returns {:error, :cart_is_empty} by definition, through a separate function clause, not through an if-check in the caller. calculate_total on an EmptyCart returns 0.0 by definition. The caller never needs to ask.

EmptyCart
   | add_item(item) // only valid transition from EmptyCart
CartWithItems
   | add_item(item) // stays CartWithItems
   | remove_item (last item) // transitions back to EmptyCart
   | calculate_total() // 0.0 on EmptyCart, sum on CartWithItems
   | checkout()
Order

"The type of the cart IS its state. There is no is_empty boolean flag. There is no null check. There is no runtime guard in any caller. If you have a CartWithItems, it has items. If you call checkout on an EmptyCart, you get an error back, handled structurally."

The state of the cart is not stored as a boolean flag or checked at runtime; it is encoded in the type of the object itself.

Summary

Summary

A Note on Classical OOP Inheritance

The instinctive object-oriented response to the shopping cart state machine would be an abstract Cart base class with EmptyCart and CartWithItems as subclasses. The problem surfaces immediately: if checkout() is declared on the base, then EmptyCart must implement it, and the only option is to throw a runtime exception. That reintroduces exactly the control complexity the type system was supposed to eliminate.

Type-driven state machines say something different. EmptyCart and CartWithItems are not specializations of a common base; they are distinct variants of a concept. checkout() does not exist on EmptyCart. The compiler enforces it.

The languages in this article handle the shared type boundary in different ways:

  • Java uses a sealed interface Cart permits EmptyCart, CartWithItems. The seal is not for code reuse; it enables exhaustive pattern matching in a switch expression and prevents subtypes outside the module. This is an algebraic sum type expressed through the interface system.
  • Scala uses a sealed trait Cart. Same principle: the compiler warns on non-exhaustive matches, and the trait carries no implementation, only the contract for exhaustive dispatch.
  • Rust is the most explicit: EmptyCart and CartWithItems are plain structs, and Cart is an enum that wraps them as variants. There is no inheritance anywhere. Exhaustive matching is enforced by the compiler on every match.
  • TypeScript expresses it as a union type: EmptyCart | CartWithItems. The two classes are entirely independent; the union is a structural constraint at the type level, not a runtime relationship.
  • Elixir has no class hierarchy at all. EmptyCart and CartWithItems are separate structs, and dispatch happens through pattern-matched function clauses. The type itself is the match guard.
  • PHP and Python, which lack sealed types, use separate final classes with no shared base. The contract is expressed through type hints, and the responsibility for exhaustiveness falls on the developer rather than the compiler.

The pattern inverts the OOP instinct. Instead of making a base class and overriding, the question becomes: what are the distinct states this thing can be in? Make each one a separate type. Attach operations only where they are valid. Express commonality through exhaustive pattern matching, not through a shared mutable interface.

Cause Traditional Approach Type-Driven Approach
State Validation flags, checked at runtime State encoded in the type; impossible to be in an invalid state
Control Conditional guards in every function body Construction either succeeds or fails; function body has no defensive branches
Code Volume Validation repeated across consumers Written once in the constructor, referenced everywhere via the type
"The type is the state. The transition is the operation. The function that receives the type can trust what the type guarantees."
Final Contract

Final OpenAPI: A Fully Type-Driven Contract

Bringing everything together: security declared through a scheme, domain types as reusable components with constraints, no primitive leakage at the boundary. The caller is identified through the Bearer token; no username parameter is needed.

  • Who can call it: only authenticated users, via Bearer token.
  • What each parameter means: types carry their own descriptions and constraints.
  • What can go wrong: 401 if the token is missing, 422 if a parameter violates a domain rule, each with a typed error response.
  • What comes back: a typed result with the calculated total.

"Every piece of knowledge that was previously buried inside the function body is now surface-level, encoded structurally in the contract itself."
